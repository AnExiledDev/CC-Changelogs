Safe Mode #

NewFlagEnv

An opt-in mode that disables all per-user and per-project customizations for a session — designed for troubleshooting when Claude Code behaves unexpectedly and you want to isolate whether a plugin, hook, or CLAUDE.md is the cause.

Usage
bashclaude --safe-mode
# or
CLAUDE_CODE_SAFE_MODE=1 claude
Details
  • Disables: CLAUDE.md, skills, plugins, plugin hooks, MCP servers (auto-discovered and claude.ai), custom agents, output styles, workflows, custom themes, and custom keybindings
  • Managed policy settings and hooks (from your organization's policy) still apply
  • The UI surfaces a warning banner so you know safe mode is active
  • Settings dialogs show “disabled in safe mode” labels next to affected options
  • To exit: remove the flag or unset the env var and restart (claude tells you which applies)
  • The --agents flag is silently ignored in safe mode

NewSetting

Windows Sandbox Network Filtering (WFP / srt-win) #

NewWindowsSandbox

Full Windows Filtering Platform (WFP) integration for the Windows sandbox via the srt-win tool. Enables per-process network isolation through Windows discriminator groups and WFP filters — analogous to the macOS sandbox's network control.

Usage
json// In settings.json (windows-specific):
{
  "windows": {
    "groupName": "my-sandbox-group",
    "groupSid": "S-1-...",
    "sublayerGuid": "..."
  }
}
Details
  • One-time install: npx sandbox-runtime windows-install (triggers one UAC prompt)
  • After install, log out and back in so the discriminator group SID enters your token
  • While the group SID is absent from your token, WFP filter-0 PERMITs all traffic — no connectivity disruption during setup
  • Default proxy port range is 60080–60089 (configurable); must match the range passed to srt-win wfp install
  • --group-sid overrides groupName lookup — use in domain environments where name resolution is unreliable
  • SRT_WIN_PATH overrides the srt-win.exe search path
  • Windows sandbox commands now run via srt-win exec instead of directly through cmd/PowerShell

LSP diagnostics Setting #

NewSettingLSP

A new diagnostics boolean on each lspServers entry controls whether publishDiagnostics notifications are automatically injected into the agent context after file edits.

Usage
json// In settings.json:
{
  "lspServers": [
    {
      "name": "my-lsp",
      "command": "...",
      "diagnostics": false
    }
  ]
}
Details
  • Defaults to true — current behavior is unchanged unless you set false
  • false keeps LSP navigation (go-to-definition, hover, completions) active while suppressing automatic error/warning injection after edits
  • Useful when diagnostics are noisy or slow the agent loop on large projects

Refusal Continuation — Silent Retry with Fallback Model #

In Development

When the model refuses a request for safety reasons, Claude Code will silently retry with a designated fallback model. If the fallback responds, the conversation continues as if nothing happened; if it also refuses, the refusal is shown normally.

⚑
Status: Feature-flagged. Gated by a server-side refusal_fallback_prompt flag that must be present in the account's beta header list.
Details
  • A refusalFallbackModelLatch tracks which fallback model was activated and the previous override, so the model is restored after the retry turn
  • fTK() returns false unless On8() (enabled) AND refusal_fallback_prompt is absent from the beta list
  • Includes latch set/clear/update functions and a refusalFallbackOccurred telemetry flag
