Organization Model Allowlist (enforceAvailableModels) #

NewOrgs

Admins can now configure an availableModels allowlist in policy settings to restrict which Claude models users and agents can use. When enforceAvailableModels is true and the list is non-empty, any attempt to use a model outside it is silently redirected to the next allowed model with a warning.

Details
  • Controlled via enforceAvailableModels and availableModels in org policy settings
  • If a preferred model (from CLI flag, SDK request, or /model switch) is not in the allowlist, Claude Code falls back to the nearest allowed model
  • modelOverrides lets admins map model aliases or families to specific allowed variants
  • Cascade trust: if a policy source exists but fails to load, enforcement is suspended and a warning is logged rather than silently defaulting to less-restricted behavior
  • Plan mode respects the allowlist — if the Opus or Haiku upgrade model is blocked, planning falls back to the resting model with a warning
  • Agent and skill model requests are also checked; blocked models trigger a notification rather than silently using an unauthorized model

Model Restriction Warning in Status Line #

NewUX

When an org policy forces a different model than you requested, a model-restricted warning now appears in the status line showing both what you asked for and what is actually being used.

Details
  • Displayed at the warning tier — higher prominence than informational indicators
  • Replaces the model-source info indicator when a restriction is active — the two never display simultaneously
  • An inline warning message is also emitted at startup when a startup-time restriction is detected

Model Restriction Notifications for Agents #

Agents

When an agent or skill requests a model blocked by the allowlist, a medium-priority warning notification now identifies which agent type made the request and what restriction was applied. Previously, blocked model requests could fail silently.

Safer Model Change Handling in SDK / Remote Sessions #

SDK

The set_model and session-initialization code paths now verify model names against the allowlist before applying them. If a model is not allowed, the change is rejected and the user sees an actionable warning instead of the session silently using an unexpected model.

availableModels Propagated from Policy Tier #

Orgs

The policy settings merge now correctly propagates availableModels from the highest-priority policy tier into the merged view used by the allowlist checker. Previously this field was not forwarded through the merge, so enforcement could not see it.

Plan Mode Model Upgrade Now Checks Allowlist #

Fix

Plan-mode model selection no longer bypasses the allowlist when upgrading from Haiku or Opus plan models. The upgrade model is now validated and falls back gracefully with a warning rather than attempting to use an unauthorized model.

Search "Plan mode: the haiku plan upgrade model is not in the availableModels allowlist"

Model Name Substring Matching Fixed #

Fix

Model name matching in the allowlist checker now uses a boundary-aware matcher that requires non-alphanumeric boundaries on both sides of the match, preventing incorrect matches where one model name is a substring of another.
