Claude Fable 5 Model #

NewModel

Fable 5 is now a first-class model in Claude Code, positioned as the most capable option for the hardest and longest-running tasks. A fable alias (and fable[1m] for 1M context) resolve to claude-fable-5.

Usage
bashclaude --model fable
claude --model claude-fable-5
Details
  • Full model IDs: claude-fable-5 (firstParty, Vertex, Foundry, Gateway), anthropic.claude-fable-5 (Bedrock/Mantle)
  • Context window: 64k default output / 128k max — same as Opus 4.8
  • Default effort level: high (same as Opus 4.8)
  • Fable 5 appears at the top of the /model picker for eligible accounts
  • A companion model claude-mythos-5 is registered internally for the safety-refusal fallback lane
  • The /model help text now mentions fable as a valid alias alongside opus and sonnet

Fable 5 Env Vars for Custom Providers #

NewEnv

Four new environment variables let BYOC and API-gateway deployments point Claude Code at a custom Fable 5 endpoint, with a fifth for Vertex region overrides.

Usage
bashexport ANTHROPIC_DEFAULT_FABLE_MODEL=my-fable-5-endpoint
export ANTHROPIC_DEFAULT_FABLE_MODEL_NAME="My Fable 5"
export ANTHROPIC_DEFAULT_FABLE_MODEL_DESCRIPTION="Custom Fable 5 model"
export ANTHROPIC_DEFAULT_FABLE_MODEL_SUPPORTED_CAPABILITIES=vision,tools
export VERTEX_REGION_CLAUDE_FABLE_5=us-east5
Details
  • ANTHROPIC_DEFAULT_FABLE_MODEL — custom Fable 5 model ID; enables the alias and fallback logic even on non-firstParty providers
  • ANTHROPIC_DEFAULT_FABLE_MODEL_NAME / _DESCRIPTION — display name and description in the model picker
  • ANTHROPIC_DEFAULT_FABLE_MODEL_SUPPORTED_CAPABILITIES — capabilities string for the model
  • VERTEX_REGION_CLAUDE_FABLE_5 — region override for Vertex AI (same pattern as existing VERTEX_REGION_CLAUDE_* vars)
  • When set on non-firstParty providers, shows a "Custom Fable model" entry in the picker in place of the default Fable 5 entry
  • New fallback hint for absent vars: "To enable automatic fallback on this provider, set ANTHROPIC_DEFAULT_FABLE_MODEL to your Fable 5 model ID and ANTHROPIC_DEFAULT_OPUS_MODEL to your Opus 4.8 model ID."

model_refusal_fallback SDK Protocol Event #

NewSDK

A new model_refusal_fallback system event is emitted over the SDK protocol whenever the primary model returns a safety refusal and the session automatically retries on a fallback model.

Details
  • Event fields: trigger: "refusal", direction: "retry" | "revert" | "sticky", original_model, fallback_model, request_id, api_refusal_category (e.g. "cyber", "bio"), api_refusal_explanation, retracted_message_uuids
  • retracted_message_uuids lists wire UUIDs of messages rolled back by the fallback — SDK consumers should evict these from transcript state on receipt
  • The supersedes field on incoming messages can now include tombstoned tool_result frames from the refused leg, not only assistant frames
  • refusalFallbackLaneEnabled and refusalFallbackSettingToggleVisible are now communicated from daemon to main process at startup, making them available to SDK consumers via init
◆
Search "model_refusal_fallback" and "retracted_message_uuids"

Server Fallback Tombstone Cancels In-Flight Tool Calls #

Fix

When a server-side refusal fallback retracts a tool dispatch mid-flight, tool calls now return a clean cancellation result instead of raising an uncaught exception.

Details
  • Cancellation message: "Tool dispatch was retracted by a server fallback; the input may be truncated."
  • The abort signal reason "server-fallback-tombstone" is checked at each phase: validate input, permission, pre-call, and post-call
◆

Fix

The retry-on-overload logic now correctly skips retrying 429 responses that contain "usage credits are required" or "extra usage is required" when the overage-disabled reason is fetch_error or org_level_disabled_until, preventing unnecessary retries in permanent overage states.
